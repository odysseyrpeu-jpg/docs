Once you have individual scripts working, you can combine them into larger automated workflows. This guide covers four common automation scenarios: chaining scripts together in sequence, wiring them into npm scripts for a familiar developer experience, running them inside a GitHub Actions CI/CD pipeline, and scheduling them with cron for time-based execution.

## Chaining scripts

The simplest way to chain scripts is to call `script run` multiple times in sequence from a parent shell script or a Makefile target. Each call blocks until the previous script exits, so you get a predictable, ordered pipeline.

<CodeGroup>

```bash scripts/pipeline.sh
#!/usr/bin/env bash
set -euo pipefail

echo "==> Step 1: Linting code..."
script run lint

echo "==> Step 2: Running tests..."
script run test

echo "==> Step 3: Building artefacts..."
script run build

echo "==> Step 4: Deploying..."
script run deploy

echo "Pipeline complete."
```

```makefile Makefile
.PHONY: pipeline lint test build deploy

pipeline: lint test build deploy

lint:
	script run lint

test:
	script run test

build:
	script run build

deploy:
	script run deploy
```

</CodeGroup>

Using `set -euo pipefail` in your parent shell script ensures the pipeline aborts immediately if any step returns a non-zero exit code, preventing later steps from running against a broken state.

## Using in npm scripts

If your project uses Node.js, you can call `script run` directly from the `scripts` block in `package.json`. This lets your team run automation through the familiar `npm run` interface without needing to remember Hiszpan Scripts command syntax.

```json package.json
{
  "name": "my-app",
  "version": "1.0.0",
  "scripts": {
    "lint": "script run lint",
    "test": "script run test",
    "build": "script run build",
    "deploy": "script run deploy",
    "pipeline": "npm run lint && npm run test && npm run build && npm run deploy"
  }
}
```

Any developer on the project can now run `npm run pipeline` without installing or learning Hiszpan Scripts separately вЂ” as long as it is listed as a project dependency or present in their `PATH`.

## CI/CD integration

The following GitHub Actions workflow runs your full pipeline on every push to `main` and on pull requests targeting `main`. Adjust the trigger conditions and steps to match your project's requirements.

```yaml .github/workflows/ci.yml
name: CI Pipeline

on:
  push:
    branches:
      - main
  pull_request:
    branches:
      - main

jobs:
  pipeline:
    runs-on: ubuntu-latest

    steps:
      - name: Checkout repository
        uses: actions/checkout@v4

      - name: Set up Node.js
        uses: actions/setup-node@v4
        with:
          node-version: "20"

      - name: Install dependencies
        run: npm ci

      - name: Install Hiszpan Scripts CLI
        run: npm install -g hiszpan-scripts

      - name: Lint
        run: script run lint

      - name: Test
        run: script run test

      - name: Build
        run: script run build
```

<Note>
  Make sure the Hiszpan Scripts CLI is available in your CI environment before you call `script run`. The example above installs it globally with `npm install -g hiszpan-scripts`. Alternatively, add it as a `devDependency` in your `package.json` and invoke it via `npx script run <name>` or through an npm script to avoid a global install step.
</Note>

## Scheduling with cron

For tasks that need to run on a time-based schedule вЂ” such as nightly data syncs, weekly report generation, or periodic cache invalidation вЂ” add a crontab entry that calls `script run` directly.

Open your crontab for editing:

```bash
crontab -e
```

Add an entry using the standard cron syntax. The example below runs the `sync-data` script every day at 02:30 AM server time:

```
# в”Њв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђ minute (0вЂ“59)
# в”‚  в”Њв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђ hour (0вЂ“23)
# в”‚  в”‚  в”Њв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђ day of month (1вЂ“31)
# в”‚  в”‚  в”‚  в”Њв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђ month (1вЂ“12)
# в”‚  в”‚  в”‚  в”‚  в”Њв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђ day of week (0вЂ“7, Sunday = 0 or 7)
# в”‚  в”‚  в”‚  в”‚  в”‚
  30 2  *  *  *  cd /path/to/your-project && script run sync-data >> logs/cron.log 2>&1
```

The `cd` command ensures Hiszpan Scripts can locate your `hiszpan.config.json` and `scriptsDir` relative to the project root. Redirecting output to a log file gives you an audit trail for scheduled runs.

<Tip>
  Write your scripts to be **idempotent** вЂ” safe to run multiple times without causing unintended side effects. For example, a database migration script should check whether a migration has already been applied before running it again. Idempotent scripts make it safe to re-run failed jobs in CI, retry cron tasks, and chain the same script into multiple workflows without risking data corruption or duplicate work.
</Tip>
