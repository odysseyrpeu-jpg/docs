Hiszpan Scripts is built around a simple mental model: you write **scripts** that do the work, a **config file** that describes how those scripts should behave, and a **CLI** that ties everything together. Whether you're automating a one-off task or orchestrating a multi-step deployment pipeline, every interaction follows the same predictable pattern — making your workflows easier to share, version, and repeat across teams and environments.

## The three pillars

<CardGroup cols={3}>
  <Card title="Scripts" icon="file-code">
    Self-contained shell or JavaScript files that live in your project. Each script encapsulates a single task and can accept parameters, read environment variables, and produce structured output.
  </Card>
  <Card title="Configuration" icon="sliders">
    The `hiszpan.config.json` file at your project root controls script discovery, default options, environment injection, and lifecycle hooks — giving you fine-grained control without touching the scripts themselves.
  </Card>
  <Card title="CLI" icon="terminal">
    The `script` command is your single entry point for running, listing, and managing scripts. It reads your config, locates the right script, and applies all configured options before execution.
  </Card>
</CardGroup>

## How scripts are discovered

When you invoke a script by name, Hiszpan Scripts looks for it in two places — in order:

1. **The `scripts/` directory** (or whatever path you set as `scriptsDir` in your config). Any file in this directory whose name matches the script name you provided is treated as a candidate.
2. **Explicit entries in `hiszpan.config.json`**. If you reference a script by an absolute or relative path in your config, that path takes precedence over the default directory scan.

This two-step lookup means you can keep most scripts co-located in a single folder for simplicity, while still pulling in shared scripts from elsewhere on your filesystem or a monorepo workspace.

## Execution flow

<Steps>
  <Step title="CLI receives command">
    You run a command such as `script run deploy`. The CLI parses the command name, any flags you pass, and any extra arguments after `--`.
  </Step>
  <Step title="Config is loaded">
    Hiszpan Scripts reads `hiszpan.config.json` from your current working directory (or the nearest ancestor directory that contains one). Config values set defaults for shell, environment variables, hooks, and verbosity.
  </Step>
  <Step title="Script is located">
    Using the resolved `scriptsDir` and any explicit path references in config, the CLI finds the file that corresponds to the script name you requested. If no match is found, it exits with a clear error message.
  </Step>
  <Step title="Script runs with options applied">
    Any configured `env` variables are injected into the process environment, `before` hooks execute first, then your script runs with any arguments you passed, and finally `after` hooks execute. Output is streamed to your terminal in real time — pass `--verbose` to see additional diagnostic information such as resolved paths and hook invocations.
  </Step>
</Steps>

## Project structure

A typical Hiszpan Scripts project looks like this:

```
my-project/
├── hiszpan.config.json         # Project-level configuration
├── scripts/
│   ├── build.sh                # Shell script: build the project
│   ├── deploy.sh               # Shell script: deploy to production
│   ├── generate-report.js      # JavaScript script: generate a report
│   └── cleanup.sh              # Shell script: remove build artifacts
├── src/
│   └── ...
├── package.json
└── README.md
```

You can nest scripts into subdirectories inside `scripts/` and reference them with a path-style name — for example, `script run db/migrate` maps to `scripts/db/migrate.sh` or `scripts/db/migrate.js`. This keeps large script collections organized without requiring any extra configuration.
